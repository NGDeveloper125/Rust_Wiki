# Feature 3: Community Articles Section

You are implementing a new feature for this project. Read this whole document first, then explore the codebase to verify every detail below, then present a plan for approval BEFORE writing any code.

## Project context (verify by reading the code)

- This repo (`Rust_Wiki`, github.com/NGDeveloper125/Rust_Wiki) is a static Rust knowledge wiki served by **GitHub Pages from the `docs/` folder on `main`**.
- The site is generated by a Rust static site generator at `tools/sitegen/` (run with `cargo run` from that directory). It reads markdown from `pages/**` and writes HTML into `docs/**`. There is no CI unless Feature 2 (Conversations) has already added one — check for `.github/workflows/`.
- Key generator modules: `main.rs` (pipeline; loads pages per section), `parse.rs` (markdown → `Page`; note it enforces the concept/syntax page section structure), `model.rs` (**`Section` enum currently has only `Syntax` and `Concepts`** — this feature likely adds a third variant or a parallel model), `markdown.rs`, `render.rs` (page + landing rendering), `nav.rs`, `search.rs` (client-side search index), `links.rs`/`bodylinks.rs` (rewrites relative `.md` links between pages into `.html` links), plus `templates/site.css` / `site.js`. No JS frameworks. Match the existing design language.
- Existing pages live at `pages/{syntax,concepts}/<subgroup>/<slug>.md` with YAML frontmatter and a fixed H2 section structure.

## The feature

An **Articles** section: long-form technical articles by the site maintainer and the community — about Rust features, libraries/crates, ecosystem topics, or opinion/approach pieces.

Requirements (decided — do not relitigate):

1. **Contribution is PR-based markdown.** An author writes an article as a markdown file and opens a pull request; the maintainer reviews for fit and validity, then merges and republishes. No web forms, no backend. The site stays 100% static.
2. Articles are **free-form prose** — unlike concept/syntax pages they must NOT be forced into the `Explanation` / `Basic usage example` / `Best practices` section structure. They need their own, much looser template: arbitrary H2/H3 structure, code blocks, links.
3. Each article has frontmatter carrying at minimum: `title`, `author` (display name), `github` (handle, for an attribution link), `date` (publication date, set at merge), `summary` (one-to-two sentences for listings), and `tags`/`topics` (small free list, e.g. `["async", "tokio", "opinion"]`).
4. **Articles index page**: lists all articles newest-first with title, author, date, tags, and summary; reachable from the site navigation and landing page consistent with existing sections. Consider (propose, don't assume) tag filtering.
5. **Article page layout**: title, byline (author linked to their GitHub profile), date, tags, then the rendered body. Styled with the existing site design language.
6. Article bodies should participate in **internal linking**: a relative link to a concept/syntax page markdown file should be rewritten to the correct `.html` URL like existing pages (`bodylinks`), and articles should be allowed to link liberally into the wiki.
7. **Search**: articles are indexed in the client-side search (title, summary, tags at minimum — propose whether full body text is indexed given index size).

## Design decisions to make WITH the user

- Directory layout: flat `pages/articles/*.md` vs. subgrouped. (Note `collect_md_files` currently expects one subgroup level — a flat layout needs a code change or a convention like `pages/articles/general/`.) Propose with tradeoffs.
- Whether articles get "related pages"-style cross-links back from concept pages, or v1 keeps linking one-directional (article → wiki). Recommend one-directional for v1.
- Whether the landing page shows recent articles.
- URL scheme: `articles/<slug>.html` vs. date-prefixed slugs.

## Also deliverable

- A **contributor guide** (extend `CONTRIBUTING.md` or the one created by Feature 1 if present): how to write and submit an article — file location, frontmatter template, style expectations, review criteria (technically accurate, code compiles, on-topic for Rust/its ecosystem, original content, civil tone), and the note that the maintainer sets/adjusts `date` at merge.
- **One real seed article** written by/for the maintainer (topic of the user's choice — ask) so the section launches non-empty and demonstrates the format.

## Acceptance criteria

- `cargo run` in `tools/sitegen` succeeds; all existing syntax/concept pages render unchanged.
- The articles index and the seed article render correctly, styled consistently, with working byline link, date, tags, and summary.
- Nav/landing integration works; search finds the seed article; internal links from the article body to wiki pages resolve to correct `.html` URLs.
- A malformed article (missing required frontmatter) produces a clear build error naming the file and the missing field — this is the maintainer's PR-review safety net.
- Contributor guide exists and matches the implemented format exactly.
- No external JS/CSS dependencies added.
