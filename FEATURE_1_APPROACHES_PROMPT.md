# Feature 1: Community "Approaches" for Best-Practice Scenarios

You are implementing a new feature for this project. Read this whole document first, then explore the codebase to verify every detail below, then present a plan for approval BEFORE writing any code.

## Project context (verify by reading the code)

- This repo (`Rust_Wiki`, github.com/NGDeveloper125/Rust_Wiki) is a static Rust knowledge wiki served by **GitHub Pages from the `docs/` folder on `main`**.
- The site is generated by a Rust static site generator at `tools/sitegen/` (run with `cargo run` from that directory). It reads markdown from `pages/**` and writes HTML into `docs/**`. There is **no CI** — the site is built locally and `docs/` is committed.
- Key generator modules: `main.rs` (pipeline), `parse.rs` (markdown → `Page`), `model.rs` (`Page`, `Scenario`, `Section` enum with `Syntax`/`Concepts`), `markdown.rs` (section/scenario splitting), `render.rs` (HTML), `nav.rs`, `search.rs`, `links.rs`/`bodylinks.rs` (internal link rewriting), plus `templates/site.css` and `templates/site.js` (copied to `docs/assets/`). No JS frameworks — plain JS only.
- Concept pages (`pages/concepts/<subgroup>/<slug>.md`) have YAML frontmatter (`title`, `area`, `embedded_support`, `groups`, `related_syntax`, `see_also`) and H2 sections: `## Explanation`, `## Basic usage example`, `## Best practices & deeper information`. The best-practices section contains `### Scenario: <name>` blocks, each parsed into a `Scenario` (body + optional rationale, see `markdown::split_scenarios` / `split_rationale`). There are also optional `(Embedded)` variants of these sections parsed into `embedded_scenarios`.

## The feature

Today, each best-practice scenario shows exactly one way to do things — the site's own recommended way. The feature: allow **multiple alternative approaches per scenario**, contributed by the community, switchable in the UI.

Requirements (these are decided — do not relitigate):

1. **The site's own approach is labeled "Classic"** and is always the default shown. Community alternatives appear as additional selectable approaches for the *same scenario*.
2. **Contribution is PR-based markdown.** Contributors add their approach by editing/adding markdown and opening a pull request; the maintainer reviews and merges. No web forms, no backend. The site stays 100% static.
3. **UI: a tab/toggle switcher per scenario** (e.g. `[Classic] [Arena-based] [Iterator-chain]`), rendered at build time with all approaches baked into the HTML and switched client-side via `site.js`. Default tab = Classic. Scenarios with only the Classic approach must render exactly as they do today (no switcher chrome for single-approach scenarios).
4. **Backward compatible:** every existing page must build and render unchanged without modification. The new markdown structure must be additive.
5. Each alternative approach must carry **attribution** (contributor name and/or GitHub handle, linkable) plus an approach title, code example, and explanation — same general shape as an existing scenario body (and it may have a rationale part like existing scenarios).
6. Applies to both normal scenarios and, where sensible, the `(Embedded)` scenario variants (they share the `Scenario` model — confirm feasibility in code and propose scope).

## Design decisions to make WITH the user before implementing

- **Markdown representation.** Propose 2–3 options and get approval. Candidates: `#### Approach: <title>` sub-blocks nested inside a `### Scenario:` block (attribution via a convention like a trailing `— by @handle` line or a small per-approach metadata line); or sibling files (e.g. `vec.approaches.md`) merged at parse time. Optimize for: easy for a contributor to add without touching the Classic content, easy to review in a PR diff, robust to parse.
- **Search indexing:** should alternative-approach text be included in the search index (`search.rs`) or only Classic? Recommend one and confirm.
- Whether attribution links to the contributor's GitHub profile.

## Also deliverable

- A **contributor guide**: a `CONTRIBUTING.md` section (or dedicated page) explaining exactly how to add an approach — file to edit, markdown format, attribution format, review criteria (code must compile, must be idiomatic-or-argued, must fit the scenario).
- At least **one real example**: add a second approach to one existing scenario on one page (author it yourself, plausibly, marked clearly) so the feature is demonstrable end to end.

## Acceptance criteria

- `cargo run` in `tools/sitegen` succeeds; all existing pages render byte-equivalent-or-visually-identical to before for pages with no alternative approaches.
- The demo page shows a working tab switcher: Classic default, clicking switches approach content, attribution visible.
- No external JS/CSS dependencies added; styling matches the existing site design language in `site.css`.
- Contributor guide exists and matches the implemented format exactly.
- Works with the existing internal-link rewriting (`bodylinks`) if an approach body links to other pages.
