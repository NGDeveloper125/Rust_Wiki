# Feature 2: Conversations Page (GitHub Discussions Mirror)

You are implementing a new feature for this project. Read this whole document first, then explore the codebase to verify every detail below, then present a plan for approval BEFORE writing any code.

## Project context (verify by reading the code)

- This repo (`Rust_Wiki`, github.com/NGDeveloper125/Rust_Wiki) is a static Rust knowledge wiki served by **GitHub Pages from the `docs/` folder on `main`**.
- The site is generated by a Rust static site generator at `tools/sitegen/` (run with `cargo run` from that directory). It reads markdown from `pages/**` and writes HTML into `docs/**`.
- **There is currently no CI** — the site is built locally and `docs/` is committed and pushed. This feature introduces the first GitHub Actions workflow.
- Key generator modules: `main.rs` (pipeline), `parse.rs`, `model.rs` (`Section` enum: `Syntax`/`Concepts`), `markdown.rs`, `render.rs`, `nav.rs`, `search.rs`, `links.rs`/`bodylinks.rs`, plus `templates/site.css` / `site.js`. No JS frameworks — plain JS only. Match the existing design language.

## The feature — architecture is DECIDED, do not relitigate

A "Conversations" page: a community forum area, backed entirely by **GitHub Discussions** with a **build-time mirror** pattern:

1. **Storage & writing:** all threads and comments live in this repo's GitHub Discussions. Users who want to start a conversation or comment click a button on the site and are **redirected to the corresponding GitHub Discussions page**. The site never accepts writes. Moderation happens with GitHub's native tools.
2. **Reading:** at build time, the generator (or a build step) fetches all discussions via the **GitHub GraphQL API** and renders them as static HTML in the site's own styling: a conversations index page (list of threads: title, author, date, comment count, category) and per-thread pages showing the full thread (post body + all comments, rendered read-only), each with an "Add a comment on GitHub →" link and the index with a "Start a conversation →" link.
3. **Freshness:** a **GitHub Actions workflow** rebuilds and republishes the site (a) on push to `main`, (b) on a **cron schedule** (propose an interval, e.g. every 6 hours), (c) on manual `workflow_dispatch`, and ideally (d) on `discussion` / `discussion_comment` webhook events so new posts appear within minutes. The page is a snapshot — near-live, not live; that is accepted.

## Hard requirements

- **Auth:** the GraphQL call authenticates with the workflow's built-in `GITHUB_TOKEN` (or a fine-grained PAT secret if `GITHUB_TOKEN` lacks Discussions read scope — verify which is needed). The token is never committed. Locally, read an optional `GITHUB_TOKEN` env var.
- **Graceful degradation:** if no token is present (typical local build) or the API call fails, the build must NOT fail — it renders the conversations page from last-known data if available, or an honest empty/placeholder state. The rest of the site always builds.
- **Security:** discussion bodies and comments are **untrusted user content**. Render their markdown safely — sanitize/escape any raw HTML to prevent XSS on the static site. This is non-negotiable; treat it as a first-class requirement.
- **CI + Pages interplay:** today `docs/` is committed on `main`. The workflow must either commit regenerated `docs/` back to `main` (needs `contents: write` and loop-protection) or switch the repo to deploy via the Pages artifact flow (`actions/upload-pages-artifact` + `actions/deploy-pages`), keeping local commits of `docs/` optional. **Present both options with tradeoffs and let the user choose** before implementing.
- Rust toolchain caching in the workflow so scheduled builds are fast/cheap.
- Nav integration: "Conversations" reachable from the site's navigation/landing page, consistent with existing sections.
- Decide (and confirm with user) whether conversations content is included in the site search index — recommend excluding it initially to keep the index stable.

## Manual steps for the user (call these out explicitly when reached)

- Enabling GitHub Discussions on the repo (Settings → Features), and creating initial categories (propose a small set, e.g. "Approaches & Idioms", "Ecosystem", "Q&A", "Site feedback").
- Creating a PAT secret if `GITHUB_TOKEN` turns out to be insufficient for reading Discussions.
- Possibly switching the Pages source setting if the artifact-deploy option is chosen.

## Design decisions to make WITH the user

- Cron frequency; whether to include the webhook-triggered rebuild in v1.
- How much of each thread to show on the index page (title-only vs. snippet).
- Whether to cache the last successful API response in-repo (e.g. a committed JSON snapshot) to power graceful degradation, or degrade to empty state only.
- Category filtering/ordering on the index page.

## Acceptance criteria

- With a valid token: build fetches real discussions and produces a styled conversations index + thread pages; all write-actions are links out to GitHub Discussions.
- Without a token: build succeeds with degraded conversations page; rest of site unaffected.
- User-authored HTML in a discussion body cannot execute as script on the rendered site (test with a `<script>` in a test discussion).
- The GitHub Actions workflow runs green: builds sitegen, regenerates the site, publishes it, on push + cron + manual dispatch.
- Site styling is consistent with existing pages; no external JS/CSS dependencies.
